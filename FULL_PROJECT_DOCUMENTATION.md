## Описание проекта

**HH.ru Student Evaluator** — это система оценки потенциальной рыночной стоимости студентов вузов на основе их академических дисциплин. Система сопоставляет дисциплины студента с реальными навыками, востребованными на рынке труда (по данным hh.ru), и рассчитывает оценочную зарплату.

### Зачем это нужно

Студенты и вузы хотят понимать, как программа обучения соотносится с реальными требованиями рынка. Система отвечает на вопрос: **«Сколько примерно может зарабатывать выпускник с такими дисциплинами?»**

### Ключевые возможности

1. **Парсинг вакансий** — автоматический сбор вакансий с hh.ru с извлечением навыков (тегов) и зарплат
2. **Семантический поиск** — сопоставление академических дисциплин с навыками hh.ru по смысловой близости (не точному совпадению)
3. **Оценка стоимости** — взвешенный расчёт потенциальной зарплаты на основе найденных навыков
4. **CRUD студентов** — создание, редактирование студентов и их дисциплин с оценками (3, 4, 5)
5. **Визуализация** — интерактивные графики влияния навыков и сходства дисциплин
6. **Гибкая настройка** — пользователь может управлять количеством навыков (top_k) и исключать нерелевантные навыки

### Потоки данных

#### Парсинг вакансий (наполнение базы данных)

1. Пользователь вводит поисковый запрос (например, «Python разработчик») и количество вакансий
2. FastAPI отправляет запрос к API hh.ru и получает вакансии
3. Из каждой вакансии извлекаются: название, URL, зарплата (salary_from, salary_to, salary_currency), теги (ключевые навыки)
4. Вакансии и теги сохраняются в PostgreSQL
5. Каждый уникальный тег преобразуется в эмбеддинг через Ollama (модель nomic-embed-text) и сохраняется в Qdrant

#### Оценка стоимости студента

1. Пользователь выбирает студента, вводит специальность, опыт работы, настраивает top_k
2. Из PostgreSQL загружаются дисциплины студента с оценками
3. Специальность семантически сопоставляется со всеми search_query вакансий — это определяет, какие вакансии релевантны
4. Для каждой дисциплины выполняется семантический поиск в Qdrant — находятся top_k ближайших навыков hh.ru
5. Для каждого найденного навыка вычисляется средняя зарплата из релевантных вакансий в PostgreSQL
6. Навыки из списка excluded_skills помечаются, но не участвуют в расчёте
7. Рассчитывается взвешенная оценка зарплаты и уверенность

## База данных (PostgreSQL)

### Модели данных

#### Vacancy (вакансия)
Хранит информацию о вакансии с hh.ru.

```python
class Vacancy:
    id: int                    # Автоинкремент PK
    hh_id: str                 # Уникальный ID на hh.ru (например, "123456")
    url: str                   # Ссылка на вакансию
    title: str                 # Название вакансии
    salary_from: int | None    # Зарплата "от" (может быть None)
    salary_to: int | None      # Зарплата "до" (может быть None)
    salary_currency: str | None # Валюта (RUR, USD, EUR и т.д.)
    experience: str | None     # Опыт работы (noExperience, between1And3, ...)
    search_query: str | None   # Какой поисковый запрос использовался для парсинга
    tags: list[Tag]            # Many-to-Many связь через vacancy_tag_association
```

#### Tag (тег/навык)
Каждый уникальный навык из вакансии.

```python
class Tag:
    id: int        # PK
    name: str      # Уникальное название навыка (например, "Python", "Docker")
    vacancies: list[Vacancy]  # Many-to-Many обратная связь
```

#### Student (студент)
Информация о студенте.

```python
class Student:
    id: int
    full_name: str               # ФИО
    group_name: str | None       # Группа (например, "CS-101")
    student_disciplines: list[StudentDiscipline]  # Связь с дисциплинами через промежуточную таблицу
```

#### Discipline (дисциплина)
Академическая дисциплина.

```python
class Discipline:
    id: int
    name: str     # Уникальное название (например, "Программирование на Python")
```

#### StudentDiscipline (связь студент-дисциплина)
Промежуточная таблица с оценкой.

```python
class StudentDiscipline:
    student_id: int      # FK → Student
    discipline_id: int   # FK → Discipline
    grade: int           # Оценка: 3, 4 или 5
```

### Связи

- **Vacancy ↔ Tag**: Many-to-Many через `vacancy_tag_association` (vacancy_id, tag_id)
- **Student ↔ Discipline**: Many-to-Many через `StudentDiscipline` (student_id, discipline_id, grade)

Все операции с БД выполняются через **асинхронный** SQLAlchemy с драйвером **asyncpg**.

## Векторная база данных (Qdrant)

### Коллекция `hh_skills`

В Qdrant хранятся векторные представления (эмбеддинги) навыков из вакансий.

**Параметры коллекции:**
- Размерность вектора: **768** (соответствует модели nomic-embed-text)
- Метрика расстояния: **косинусное сходство** (Cosine distance)
- Каждая точка в коллекции содержит:
  - `id` — хеш имени навыка
  - `vector` — 768-мерный эмбеддинг
  - `payload` — `{ "name": "Python" }` — название навыка

### Операции

1. **Upsert** — при парсинге вакансии каждый новый тег преобразуется в эмбеддинг и добавляется в коллекцию (если уже существует — обновляется)
2. **Search** — для дисциплины генерируется эмбеддинг и ищутся top_k ближайших навыков по косинусному расстоянию

### Клиент (vector_store.py)

```python
class VectorStore:
    async def init_collections()         # Создать коллекцию hh_skills
    async def upsert_skill(name, vector) # Добавить/обновить навык
    async def search_similar_skills(     # Найти ближайшие навыки
        text: str,
        top_k: int = 5,
        score_threshold: float = 0.6
    ) -> list[dict]  # [{"name": "Python", "score": 0.92}, ...]
```

## Эмбеддинги (Ollama)

### Модель: nomic-embed-text

**nomic-embed-text** — мультиязычная модель для генерации эмбеддингов текста. Ключевые характеристики:

- **Размерность**: 768
- **Мультиязычность**: поддерживает русский и английский текст, что важно для сопоставления русских названий дисциплин с тегами (часто на английском)
- **Контекстное окно**: 8192 токена
- Работает **локально** через Ollama без отправки данных во внешние API

### Клиент (embeddings.py)

```python
class EmbeddingService:
    async def get_embedding(text: str) -> list[float]           # Один текст → вектор
    async def get_embeddings_batch(texts: list[str]) -> list[list[float]]  # Пакетная генерация
    async def ensure_model_loaded()  # Проверка/загрузка модели в Ollama
```

Запросы к Ollama идут через httpx на `http://ollama:11434/api/embeddings`.

## Алгоритм оценки стоимости студента

Это ключевой алгоритм системы, реализованный в файле `app/valuation.py`.

### Входные данные

| Параметр | Тип | Описание |
|----------|-----|----------|
| `disciplines` | `list[DisciplineWithGrade]` | Список дисциплин студента с оценками (3, 4, 5) |
| `specialty` | `str` | Специальность для фильтрации вакансий (например, «Python разработчик») |
| `experience` | `str \| None` | Фильтр по опыту (noExperience, between1And3, between3And6, moreThan6) |
| `top_k` | `int` (1–20) | Сколько ближайших навыков искать на каждую дисциплину. По умолчанию 5 |
| `excluded_skills` | `list[str] \| None` | Навыки, которые пользователь хочет исключить из расчёта |

### Пошаговый алгоритм

#### Шаг 1: Фильтрация вакансий по специальности

Функция `get_matching_search_queries()` определяет, какие вакансии релевантны для данной специальности.

1. Из PostgreSQL загружаются все уникальные значения `search_query` из таблицы вакансий
2. Для специальности и каждого search_query генерируются эмбеддинги через Ollama
3. Вычисляется косинусное сходство между эмбеддингами
4. Отбираются search_query с cosine_similarity >= 0.7 (порог `SPECIALTY_SIMILARITY_THRESHOLD`)

Пример: если specialty = «Python разработчик», то могут совпасть search_query: «python developer», «python backend», «Python» и т.д.

#### Шаг 2: Семантический поиск навыков для каждой дисциплины

Для каждой дисциплины студента:

1. Название дисциплины преобразуется в эмбеддинг через Ollama
2. В Qdrant ищутся `top_k` ближайших навыков (по косинусному расстоянию)
3. Отсеиваются результаты с similarity < 0.6 (порог `similarity_threshold`)

Результат: список пар `(навык_hh, similarity_score)`.

Пример для дисциплины «Программирование на Python»:
```
[("Python", 0.92), ("Django", 0.78), ("FastAPI", 0.75), ("Flask", 0.71), ("Backend", 0.65)]
```

#### Шаг 3: Получение зарплатной статистики

Для каждого найденного навыка из PostgreSQL извлекаются:
- **avg_salary** — средняя зарплата вакансий с этим тегом
- **vacancy_count** — количество таких вакансий

Учитываются только вакансии, которые:
- Содержат данный тег
- Имеют зарплату в рублях (`salary_currency = "RUR"`)
- Соответствуют отфильтрованным search_query (из шага 1)
- Соответствуют фильтру по опыту (если указан)

Навыки с менее чем 3 вакансиями (`MIN_TAG_COUNT = 3`) считаются статистически ненадёжными и не участвуют в расчёте (но отображаются в UI с пометкой ⚠️).

#### Шаг 4: Обработка исключённых навыков

Если пользователь исключил навыки через `excluded_skills`:
- Каждый навык проверяется (case-insensitive) на принадлежность к списку excluded_skills
- Исключённые навыки помечаются `excluded: true`
- Они остаются в ответе API для отображения, но их вес НЕ учитывается в формуле
- В UI они показываются затемнёнными и перечёркнутыми, с кнопкой восстановления

Это полезно, когда семантический поиск нашёл нерелевантный навык (например, «Коммуникации» для дисциплины «Компьютерные сети»).

#### Шаг 5: Расчёт взвешенной оценки

Формула итоговой оценки зарплаты:

```
                    Σ (avg_salary_i × weight_i)
estimated_salary = ─────────────────────────────
                         Σ (weight_i)
```

Вес каждого навыка:

```
weight_i = similarity_i × log(1 + vacancy_count_i) × grade_coeff_i
```

Где:
- `similarity_i` — косинусное сходство дисциплины и навыка (от 0 до 1)
- `log(1 + vacancy_count_i)` — логарифмированная частотность навыка (сглаживает выбросы)
- `grade_coeff_i` — коэффициент оценки студента по дисциплине

Коэффициенты оценок:

| Оценка | Коэффициент | Смысл |
|--------|-------------|-------|
| 5 (отлично) | 1.0 | Полный вклад навыка |
| 4 (хорошо) | 0.85 | Немного сниженный вклад |
| 3 (удовлетворительно) | 0.75 | Значительно сниженный вклад |

Навыки, помеченные как исключённые (`excluded: true`), а также навыки с менее чем 3 вакансиями, НЕ участвуют в этой формуле.

#### Шаг 6: Расчёт уверенности

```
confidence = matched_disciplines / total_disciplines
```

- `matched_disciplines` — количество дисциплин, для которых нашёлся хотя бы один валидный навык (не исключённый, >= 3 вакансий, similarity >= порог)
- Значение от 0.0 до 1.0

### Выходные данные

```python
@dataclass
class ValuationResult:
    estimated_salary: float | None  # Оценочная зарплата (None если нет данных)
    confidence: float               # Уверенность (0..1)
    skill_matches: list[SkillMatch] # Все найденные навыки с метаданными
    total_disciplines: int          # Всего дисциплин
    matched_disciplines: int        # Дисциплин с найденными навыками

@dataclass
class SkillMatch:
    discipline: str       # Название дисциплины
    skill_name: str       # Название навыка hh.ru
    similarity: float     # Косинусное сходство
    avg_salary: float | None  # Средняя ЗП по вакансиям с этим навыком
    vacancy_count: int    # Количество вакансий
    grade: int            # Оценка студента (3, 4, 5)
    grade_coeff: float    # Коэффициент оценки
    excluded: bool        # Исключён ли навык из расчёта пользователем
```

## Фронтенд (SPA)

Фронтенд реализован как один HTML-файл (`app/static/index.html`) с встроенными CSS и JavaScript. Используется темная тема в стиле GitHub Dark.

### Вкладки (табы)

1. **📊 Оценка студента** — главная вкладка
   - Выбор студента из выпадающего списка
   - Ввод специальности
   - Выбор опыта работы
   - **Слайдер top_k** (1–20) — управляет количеством навыков на дисциплину
   - Кнопка «Оценить стоимость»
   - Результаты: три карточки (оценочная ЗП, уверенность, совпавшие дисциплины)
   - График «Влияние навыков на стоимость» (horizontal bar chart)
   - График «Сходство дисциплин с навыками» (radar chart)
   - Таблица детальной разбивки с кнопками 🗑️ для исключения навыков
   - Блок «Исключённые навыки» — фишки с кнопкой ✕ для возврата

2. **👩‍🎓 Студенты** — CRUD
   - Форма создания/редактирования студента
   - Добавление дисциплин с оценками (кнопки 3/4/5)
   - Список студентов с возможностью редактирования
   - Отдельная форма для быстрого добавления дисциплин существующему студенту

3. **🔍 Парсинг вакансий** — сбор данных
   - Ввод запроса, количества, опыта
   - Лог парсинга в реальном времени
   - Графики топ-навыков после парсинга

4. **🏷️ Навыки и теги** — аналитика
   - Статистика по сохранённым тегам
   - Doughnut chart распределения навыков
   - Таблица всех навыков

### Механизм исключения навыков на фронтенде

Работает так:

1. Пользователь оценивает студента → получает таблицу навыков
2. В каждой строке таблицы есть кнопка 🗑️ (исключить)
3. При клике навык добавляется в массив `excludedSkills` (JavaScript)
4. Автоматически отправляется повторный запрос `/evaluate` с `excluded_skills=...`
5. Результаты перерисовываются, исключённый навык показывается затемнённым с кнопкой ♻️ (восстановить)
6. Над таблицей появляется блок «🚫 Исключённые навыки» с фишками — каждая фишка содержит название навыка и кнопку ✕ для возврата
7. При восстановлении навык удаляется из `excludedSkills`, и снова отправляется запрос пересчёта

Используемые библиотеки: Chart.js 4.4.7 (CDN).

## Парсинг вакансий с hh.ru

Файл `app/parser.py` реализует парсер вакансий.

### Процесс парсинга

1. Запрос к `https://api.hh.ru/vacancies` с параметрами `text`, `per_page`, `experience`
2. Для каждой вакансии запрос деталей: `https://api.hh.ru/vacancies/{id}`
3. Извлечение данных: title, url, salary_from, salary_to, salary_currency, key_skills
4. Сохранение вакансии в PostgreSQL (upsert по hh_id)
5. Для каждого тега:
   - Создание записи Tag в PostgreSQL (если не существует)
   - Привязка к вакансии через vacancy_tag_association
   - Генерация эмбеддинга через Ollama
   - Upsert в Qdrant (коллекция hh_skills)

### Важные детали

- Парсер работает с **пагинацией** API hh.ru (по 20 вакансий на страницу)
- Поддерживает **фильтр по опыту** (noExperience, between1And3, between3And6, moreThan6)
- При парсинге `search_query` записывается в каждую вакансию — это используется позже для фильтрации
- Зарплатная статистика учитывает **только RUB** (`salary_currency = "RUR"`)
