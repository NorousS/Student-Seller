# PRD: Воспроизводимый запуск проекта после чистого git clone

## 1. Introduction / Overview

Сейчас проект запускается локально у владельца репозитория через `docker-compose`, но в чистом клоне (`git clone` в новую папку) сборка/подъём окружения не повторяется стабильно. Это признак проблемы с версионированием артефактов и/или правилами игнорирования.

Цель: обеспечить предсказуемый сценарий для нового разработчика — после чистого клона проект должен запускаться через `docker-compose up --build` без ручных правок.

## 2. Goals

- Обеспечить успешный запуск в чистом клоне через `docker-compose up --build`.
- Исправить только подтверждённые проблемные правила в `.gitignore`.
- Выполнить `commit` и `push` в текущую ветку с минимальным, целевым диффом.
- Подтвердить стабильность запуском backend-тестов после подъёма окружения.
- Не вносить изменения в бизнес-логику приложения.

## 3. User Stories

### US-001: Зафиксировать и локализовать проблему воспроизводимости
**Description:** Как разработчик, я хочу воспроизвести сбой в чистом клоне, чтобы точно определить, какие файлы/паттерны игнорирования ломают запуск.

**Acceptance Criteria:**
- [ ] Репозиторий клонируется в отдельную папку.
- [ ] Выполняется `docker-compose up --build` и фиксируется фактический сбой (если он есть) с конкретным шагом/файлом.
- [ ] Сформирован краткий список "что отсутствует или конфликтует из-за версионирования".
- [ ] Команда и ошибка воспроизводимы повторно.

### US-002: Исправить `.gitignore` точечно по подтверждённой причине
**Description:** Как разработчик, я хочу скорректировать `.gitignore` только по подтверждённым путям, чтобы не расширять изменения вне проблемы.

**Acceptance Criteria:**
- [ ] В `.gitignore` изменены только правила, напрямую влияющие на воспроизводимость сборки.
- [ ] Не добавлены потенциально секретные файлы (например приватные `.env` с реальными токенами).
- [ ] Дифф `.gitignore` минимален и объясним.
- [ ] После изменений проблемный сценарий больше не зависит от локальных незакоммиченных артефактов.

### US-003: Зафиксировать изменение в git и отправить в remote
**Description:** Как владелец проекта, я хочу получить commit+push в текущую ветку, чтобы исправление было доступно для повторного клона.

**Acceptance Criteria:**
- [ ] Выполнен commit только с релевантными изменениями по задаче.
- [ ] Commit отправлен (`push`) в текущую ветку.
- [ ] Рабочее дерево после commit чистое (кроме заранее известных нерелевантных локальных файлов, если они были до начала задачи).
- [ ] Сообщение commit однозначно отражает фиксацию проблемы воспроизводимости.

### US-004: Подтвердить исправление в новом чистом клоне
**Description:** Как новый разработчик, я хочу повторить процесс с нуля, чтобы убедиться, что проект запускается без ручных действий.

**Acceptance Criteria:**
- [ ] После push выполнен новый `git clone` в отдельную папку.
- [ ] В новом клоне `docker-compose up --build` завершается без ошибок сборки.
- [ ] Сервисы `app`, `db`, `qdrant`, `ollama` стартуют и остаются в состоянии running/healthy согласно конфигурации.
- [ ] Логи не содержат фатальных ошибок на старте.

### US-005: Прогнать backend-тесты в контексте исправления
**Description:** Как команда разработки, мы хотим проверить backend-тесты после фикса, чтобы исключить регрессии.

**Acceptance Criteria:**
- [ ] После успешного старта окружения запущены backend-тесты.
- [ ] Результат тестов сохранён в отчёте выполнения задачи (кол-во passed/failed/skipped).
- [ ] При падении тестов явно выделены новые ошибки, связанные/не связанные с текущим изменением.

## 4. Functional Requirements

- FR-1: Система выполнения работ должна воспроизвести проблему в чистом клоне до внесения правок.
- FR-2: Изменения в `.gitignore` должны быть строго ограничены подтверждённой причиной сбоя.
- FR-3: Нельзя добавлять в версионирование секретные/локально-чувствительные данные.
- FR-4: Исправление должно быть оформлено commit и отправлено в текущую ветку (`push`).
- FR-5: После `push` должен быть выполнен новый `git clone` в отдельную директорию.
- FR-6: В новом клоне команда `docker-compose up --build` должна завершаться успешно.
- FR-7: После сборки и старта окружения должны запускаться backend-тесты для проверки отсутствия регрессий.
- FR-8: Итог должен включать фактическое подтверждение воспроизводимости (что именно запускалось и что получено).

## 5. Non-Goals (Out of Scope)

- Изменение бизнес-логики backend/frontend.
- Рефакторинг Docker-конфигурации, не связанный напрямую с проблемой.
- Изменения CI/CD пайплайнов.
- Добавление новых фич или оптимизаций вне воспроизводимости запуска.

## 6. Design Considerations (Optional)

Для задачи не требуются UI/UX изменения.

## 7. Technical Considerations (Optional)

- Проект использует multi-stage Docker build: frontend собирается внутри Docker-образа и копируется в backend-слой.
- Приоритет — согласованность между содержимым репозитория и ожиданиями Docker-сборки.
- Любые изменения `.gitignore` должны сохранять текущие практики безопасности (не раскрывать секреты).

## 8. Success Metrics

- Чистый клон запускается через `docker-compose up --build` без ручных дополнительных шагов.
- Исправление ограничено `.gitignore` и сопутствующим минимальным git-процессом.
- Backend-тесты после исправления проходят без новых регрессий.
- Любой участник команды может повторить шаги и получить тот же результат.

## 9. Open Questions

- Нужна ли дополнительная smoke-проверка API после старта контейнеров, помимо факта успешного `up --build` и backend-тестов?
- Нужно ли фиксировать артефакт-лог (например, отдельный файл) или достаточно отчёта в сообщении по задаче?
