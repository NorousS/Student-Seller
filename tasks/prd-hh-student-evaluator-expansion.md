# PRD: Масштабное расширение HH.ru Student Evaluator для работодателей

## 1. Introduction / Overview

Текущая версия платформы оценивает студентов, но недостаточно хорошо решает задачу быстрого найма для работодателей с срочной кадровой потребностью.  
Нужен полноценный продуктовый слой: SEO-лендинг с сильным first screen, ролевой доступ для партнеров/непартнеров, представление студента на языке бизнеса (укрупненные блоки компетенций), а также более обоснованная математическая модель оценки рыночной стоимости.

Цель релиза — сделать систему понятной и ценной для работодателя с сохранением приватности студента и прозрачности оценки.

## 2. Goals

- Увеличить конверсию работодателя из первого экрана в действие «Пригласить на собеседование».
- Показывать 5 лучших анонимных профилей с рыночной ценой и кратким competency-профилем.
- Реализовать двухуровневый доступ работодателей (партнер / сторонняя организация).
- Перевести отображение дисциплин в укрупненные блоки компетенций с агрегированными метриками.
- Усилить математическое ядро оценки (статистика/ML + рыночная калибровка + объяснимость).
- Сохранить контроль персональных контактов студента до принятия запроса.

## 3. User Stories

### US-001: Показ 5 анонимных профилей на первом экране лендинга
**Description:** Как работодатель, я хочу сразу видеть лучших кандидатов и их потенциальную стоимость, чтобы быстро понять ценность платформы.

**Acceptance Criteria:**
- [ ] На первом экране отображаются ровно 5 карточек студентов, отсортированных по внутреннему рейтингу.
- [ ] В карточке есть: анонимное фото, потенциальная зарплата, краткое описание компетенций.
- [ ] В каждой карточке есть кнопка «Пригласить на собеседование».
- [ ] Для неавторизованного пользователя не показываются персональные контакты.
- [ ] Typecheck/lint passes.
- [ ] Verify in browser using dev-browser skill.

### US-002: Ролевой запуск сценария «Пригласить на собеседование»
**Description:** Как система, я хочу запускать разный сценарий по кнопке приглашения в зависимости от типа работодателя, чтобы корректно применять бизнес-правила монетизации.

**Acceptance Criteria:**
- [ ] При клике система определяет роль и тип работодателя (партнер/непартнер).
- [ ] Для партнера запускается создание запроса студенту без paywall-модалки.
- [ ] Для непартнера открывается paywall-модалка с вариантами монетизации.
- [ ] Во всех вариантах событие клика логируется в аналитике.
- [ ] Typecheck/lint passes.
- [ ] Verify in browser using dev-browser skill.

### US-003: SEO-оптимизация лендинга
**Description:** Как продуктовая команда, мы хотим SEO-оптимизированный лендинг, чтобы получать релевантный органический трафик работодателей.

**Acceptance Criteria:**
- [ ] Для лендинга заданы title, description, OpenGraph и canonical.
- [ ] Есть корректная семантическая разметка заголовков (H1/H2/H3) и индексируемый контент.
- [ ] robots/sitemap обновлены и включают лендинг.
- [ ] Core Web Vitals не деградируют относительно текущего baseline.
- [ ] Typecheck/lint passes.
- [ ] Verify in browser using dev-browser skill.

### US-004: Разделение работодателей на партнеров и сторонние организации
**Description:** Как админ, я хочу помечать работодателей статусом партнерства, чтобы управлять доступом к функционалу.

**Acceptance Criteria:**
- [ ] В профиле работодателя хранится признак партнерства и его источник (договор/ручная валидация).
- [ ] API возвращает признак партнерства в профиле работодателя.
- [ ] Изменение статуса партнерства логируется в audit trail.
- [ ] Непартнер не получает партнерские привилегии без смены статуса.
- [ ] Typecheck/lint passes.

### US-005: Paywall-модалка для сторонних организаций
**Description:** Как непартнерский работодатель, я хочу видеть понятные варианты доступа, чтобы принять решение об оплате или партнерстве.

**Acceptance Criteria:**
- [ ] При попытке приглашения открывается модалка с минимум двумя CTA: «Оплатить доступ» и «Заключить договор с вузом».
- [ ] Без выбора CTA запрос на собеседование не создается.
- [ ] После успешной оплаты/инициации договора пользователь переводится в соответствующий сценарий.
- [ ] Все события модалки передаются в аналитику.
- [ ] Typecheck/lint passes.
- [ ] Verify in browser using dev-browser skill.

### US-006: Контакты студента открываются только после принятия запроса
**Description:** Как студент, я хочу, чтобы мои контакты были защищены до моего согласия, чтобы сохранять контроль над персональными данными.

**Acceptance Criteria:**
- [ ] Телефон и email скрыты для всех работодателей до статуса «Запрос принят студентом».
- [ ] При отклонении запроса контакты остаются скрытыми.
- [ ] При принятии запроса контакты раскрываются только инициатору запроса.
- [ ] Попытка несанкционированного получения контактов возвращает ошибку авторизации.
- [ ] Typecheck/lint passes.

### US-007: Расширенный профиль для партнеров
**Description:** Как партнерский работодатель, я хочу видеть больше профессиональных данных студента, чтобы принимать решение о собеседовании точнее.

**Acceptance Criteria:**
- [ ] Партнер видит достижения, курсовые проекты, укрупненные блоки знаний и даты готовности к работе.
- [ ] Непартнер видит только ограниченную анонимизированную версию.
- [ ] Полная академическая ведомость не показывается работодателю ни в одном режиме.
- [ ] Доступ проверяется на backend, а не только на frontend.
- [ ] Typecheck/lint passes.

### US-008: Автоматическая категоризация дисциплин в компетентностные блоки
**Description:** Как система, я хочу группировать дисциплины в блоки компетенций через LLM, чтобы показывать работодателю понятную структуру навыков.

**Acceptance Criteria:**
- [ ] Для каждой дисциплины определяется категория (competence block) через Ollama.
- [ ] Результат категоризации сохраняется и переиспользуется без повторной генерации при отсутствии изменений.
- [ ] Для неклассифицированных дисциплин применяется контролируемый fallback-класс с логированием.
- [ ] Доступен механизм ручной корректировки категории админом.
- [ ] Typecheck/lint passes.

### US-009: Расчет агрегированных метрик по блоку компетенций
**Description:** Как работодатель, я хочу видеть сводные показатели по каждому блоку, чтобы быстро оценивать силу кандидата.

**Acceptance Criteria:**
- [ ] Для блока считается средний балл по формуле G_avg = sum(grade_i)/n.
- [ ] Для блока считается агрегированная рыночная ценность из дисциплин/тегов.
- [ ] Считаются strong points: число «отлично» и число релевантных курсовых.
- [ ] Метрики пересчитываются при изменении дисциплин, оценок или весов модели.
- [ ] Typecheck/lint passes.

### US-010: Карточки компетенций в Employer Panel
**Description:** Как работодатель, я хочу видеть карточки компетенций вместо таблицы дисциплин, чтобы быстрее понимать профиль кандидата.

**Acceptance Criteria:**
- [ ] В Employer Panel таблица дисциплин заменена на grid карточек компетенций.
- [ ] В карточке есть: название блока, уровень владения, топ-3 рыночных тега, краткая выжимка достижений.
- [ ] Карточки адаптивны для desktop и tablet.
- [ ] Данные карточек соответствуют backend-агрегации без расхождений.
- [ ] Typecheck/lint passes.
- [ ] Verify in browser using dev-browser skill.

### US-011: Матрица видимости данных по ролям
**Description:** Как владелец продукта, я хочу строгую матрицу видимости, чтобы каждая роль видела только допустимый уровень деталей.

**Acceptance Criteria:**
- [ ] Студент и админ видят полный список дисциплин, оценки и инструменты управления профилем.
- [ ] Работодатель (анонимный/базовый доступ) видит только блоки, средний уровень и стоимость.
- [ ] Работодатель-партнер видит расширенную детализацию по компетенциям без личных контактов до согласия.
- [ ] Матрица покрыта API-тестами на все роли.
- [ ] Typecheck/lint passes.

### US-012: Усиленная модель рыночной оценки
**Description:** Как команда продукта, мы хотим использовать продвинутую статистическую/ML-модель, чтобы повысить обоснованность оценки стоимости студента.

**Acceptance Criteria:**
- [ ] Модель учитывает similarity, рыночную частоту навыка, академическую успеваемость и временную актуальность рынка.
- [ ] Поддерживается калибровка по историческим данным вакансий.
- [ ] Модель возвращает не только итоговую оценку, но и разложение вклада факторов.
- [ ] Расчет укладывается в целевой SLA endpoint-а оценки.
- [ ] Typecheck/lint passes.

### US-013: Обоснование уникальности технологического стека
**Description:** Как команда проекта, мы хотим формализовать уникальность стека, чтобы обосновать технологическое преимущество решения.

**Acceptance Criteria:**
- [ ] В системе зафиксирован набор применяемых технологий (PostgreSQL + Qdrant + Ollama + семантический матчинг).
- [ ] Для оценки сохраняются трассировочные данные: какие компетенции и теги привели к итоговой цене.
- [ ] Есть экспорт объяснения оценки для демонстрации заинтересованным сторонам.
- [ ] Typecheck/lint passes.

### US-014: Готовность к трудоустройству в профиле
**Description:** Как работодатель, я хочу видеть дату готовности студента к выходу на работу, чтобы планировать найм.

**Acceptance Criteria:**
- [ ] В профиле отображается дата готовности к работе (например, дата выдачи диплома).
- [ ] Поле можно поддерживать как автоматически (из учебного плана), так и вручную админом.
- [ ] Дата валидируется и не может быть в некорректном формате.
- [ ] Typecheck/lint passes.
- [ ] Verify in browser using dev-browser skill.

## 4. Functional Requirements

- FR-1: Система должна показывать на первом экране лендинга ровно 5 анонимных профилей лучших студентов.
- FR-2: Карточка студента должна включать анонимное фото, потенциальную зарплату и краткий competency-профиль.
- FR-3: Каждая карточка должна содержать CTA «Пригласить на собеседование».
- FR-4: При нажатии CTA система должна определять тип работодателя и выбирать соответствующий сценарий.
- FR-5: Для партнерского работодателя приглашение должно создаваться без paywall-модалки.
- FR-6: Для непартнерского работодателя система должна показывать paywall с вариантами монетизации.
- FR-7: Система не должна раскрывать телефон/email студента до принятия запроса студентом.
- FR-8: Система должна реализовать хранение и админ-управление статуса партнерства работодателя.
- FR-9: Профиль работодателя должен получать признак партнерства через API.
- FR-10: Дисциплины студента должны автоматически группироваться в компетентностные блоки.
- FR-11: Для каждой дисциплины должна храниться категория компетенции и возможность ручной коррекции.
- FR-12: Для каждого блока компетенций должен считаться средний балл, рыночная ценность и strong points.
- FR-13: Employer Panel должна показывать grid карточек компетенций вместо таблицы дисциплин.
- FR-14: Карточка компетенции должна включать название блока, уровень, топ-3 тега рынка и достижения.
- FR-15: Система должна реализовать матрицу видимости данных по ролям (admin/student/employer/employer-partner).
- FR-16: Модель оценки должна учитывать рыночные и академические факторы, а не только базовую similarity.
- FR-17: API оценки должен возвращать объяснение вклада факторов в итоговую стоимость.
- FR-18: Лендинг должен быть SEO-оптимизирован (метаданные, индексируемая структура, sitemap/robots).
- FR-19: Все ключевые события воронки (просмотр карточки, клик CTA, paywall, приглашение) должны логироваться.
- FR-20: Профиль студента должен содержать и отображать дату готовности к трудоустройству.

## 5. Non-Goals (Out of Scope)

- Подготовка курсовой работы, методичек и академических отчетов как deliverable этого релиза.
- Полный редизайн всей платформы за пределами лендинга и Employer Panel.
- Публикация production-хостинга и DevOps-пайплайнов за пределами текущей инфраструктуры.
- Автоматическое принятие приглашений от работодателя без участия студента.
- Открытие прямых контактов студентам/работодателям вне утвержденного workflow.

## 6. Design Considerations

- Основной UX-фокус — скорость принятия решения работодателем в первые 30–60 секунд.
- Визуальный язык карточек должен подчеркивать сильные стороны, а не показывать «сырые» оценки.
- Для paywall-модалки нужен короткий, коммерчески понятный copy без перегрузки деталями.
- Компоненты должны переиспользовать существующий дизайн-системный подход проекта.

## 7. Technical Considerations

- Backend: FastAPI + SQLAlchemy async; расширение endpoint-ов работодателя и оценки.
- Data: PostgreSQL для профилей/ролей/событий, Qdrant для векторного поиска, Ollama для embeddings/категоризации.
- Для группировки дисциплин требуется отдельный слой агрегации (например, `aggregate_by_competence()` в evaluation flow).
- Нужна миграция данных для хранения категории дисциплины и статуса партнерства работодателя.
- Все ролевые ограничения должны проверяться на backend-уровне.

## 8. Success Metrics

- Не менее 70% работодателей, попавших на лендинг, взаимодействуют хотя бы с одной карточкой.
- Не менее 25% взаимодействий с карточкой переходят в клик «Пригласить на собеседование».
- Время понимания профиля кандидата работодателем сокращено минимум на 40% (по UX-замерам).
- Не менее 95% запросов оценки укладываются в целевой SLA без деградации качества.
- Доля успешных классификаций дисциплин в компетентностные блоки не ниже 90%.
