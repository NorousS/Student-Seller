# PRD: Редизайн лендинга и маршрутизации

## Введение

Переработка главной страницы проекта HH Parser. Сейчас `/` делает redirect на `/landing`, где отображается минималистичная страница с топ-5 студентов и одной CTA-кнопкой. Нужно:
1. Рендерить полноценный лендинг прямо на `/` (без redirect)
2. Сделать красивый лендинг с секциями: hero → преимущества → топ-студенты → CTA
3. Добавить кнопку «Войти» в хедер лендинга и в hero-секцию
4. Протестировать весь проект через Playwright E2E в Docker

## Цели

- Первое впечатление пользователя — красивый маркетинговый лендинг на `/`
- Навигация «Войти» доступна из хедера и hero-секции
- Навигация «Зарегистрироваться» доступна из hero-секции и footer CTA
- Секция «Топ-студенты» визуально улучшена
- Весь существующий функционал проекта не сломан (регрессионный E2E)

## Пользовательские истории

### US-001: Лендинг рендерится на `/`
**Описание:** Как неавторизованный пользователь, я хочу видеть лендинг при открытии `/`, а не быть перенаправленным на `/landing`.

**Критерии приёмки:**
- [ ] Компонент `LandingPage` рендерится на маршруте `/` для неавторизованных пользователей
- [ ] Маршрут `/landing` остаётся рабочим (redirect на `/` или рендерит тот же компонент)
- [ ] Авторизованные пользователи при переходе на `/` по-прежнему перенаправляются в свой кабинет (student → `/student`, employer → `/employer`, admin → `/admin`)
- [ ] Typecheck проходит (`tsc -b`)
- [ ] Проверить в браузере через Playwright

### US-002: Хедер лендинга с навигацией
**Описание:** Как пользователь, я хочу видеть навигационную панель сверху лендинга с логотипом и кнопками «Войти» / «Зарегистрироваться».

**Критерии приёмки:**
- [ ] Хедер лендинга отображается только для неавторизованных пользователей на лендинге (не показывается в залогиненном состоянии — там другой хедер)
- [ ] Хедер содержит: логотип/название проекта слева, кнопки «Войти» и «Зарегистрироваться» справа
- [ ] Кнопка «Войти» ведёт на `/login`
- [ ] Кнопка «Зарегистрироваться» ведёт на `/register`
- [ ] Хедер фиксирован сверху (sticky/fixed) с полупрозрачным фоном
- [ ] Typecheck проходит
- [ ] Проверить в браузере через Playwright

### US-003: Hero-секция
**Описание:** Как пользователь, я хочу видеть привлекательную hero-секцию с описанием платформы и кнопками действий.

**Критерии приёмки:**
- [ ] Заголовок: крупный, привлекательный (с градиентом или акцентным цветом)
- [ ] Подзаголовок: краткое описание платформы (1-2 предложения)
- [ ] Две кнопки: «Войти» (вторичная/outline) и «Зарегистрироваться» (primary)
- [ ] Кнопка «Войти» ведёт на `/login`, «Зарегистрироваться» ведёт на `/register`
- [ ] Hero визуально выделяется (градиентный фон, большие отступы, центрирование)
- [ ] Typecheck проходит
- [ ] Проверить в браузере через Playwright

### US-004: Секция «Преимущества платформы»
**Описание:** Как посетитель лендинга, я хочу видеть список ключевых преимуществ платформы, чтобы понять её ценность.

**Критерии приёмки:**
- [ ] Секция с 3-4 карточками преимуществ (иконка + заголовок + описание)
- [ ] Примеры преимуществ: «ИИ-оценка навыков», «Реальные данные с hh.ru», «Анонимный подбор», «Быстрый доступ к кандидатам»
- [ ] Карточки в сетке (2-3 колонки на десктопе, 1 на мобильном)
- [ ] Используется существующий стиль `.card` с улучшениями (hover-эффекты, иконки)
- [ ] Typecheck проходит
- [ ] Проверить в браузере через Playwright

### US-005: Улучшенная секция «Топ-студенты»
**Описание:** Как посетитель лендинга, я хочу видеть красиво оформленную секцию с топ-кандидатами.

**Критерии приёмки:**
- [ ] Секция использует существующий API `GET /api/v1/landing/top-students`
- [ ] Карточки студентов визуально улучшены (hover-анимации, лучшая типографика)
- [ ] Если нет данных — секция скрыта или показывает плейсхолдер
- [ ] Кнопка «Пригласить» по-прежнему ведёт на `/login` для неавторизованных
- [ ] Typecheck проходит
- [ ] Проверить в браузере через Playwright

### US-006: Footer CTA-секция
**Описание:** Как посетитель лендинга, я хочу видеть финальный призыв к действию внизу страницы.

**Критерии приёмки:**
- [ ] CTA-блок с заголовком «Готовы начать?» или аналогичным
- [ ] Кнопка «Зарегистрироваться» (primary, крупная)
- [ ] Опционально: ссылка «Уже есть аккаунт? Войти»
- [ ] Typecheck проходит
- [ ] Проверить в браузере через Playwright

### US-007: E2E тестирование всего проекта
**Описание:** Как разработчик, я хочу убедиться, что все существующие и новые функции работают корректно в Docker-окружении.

**Критерии приёмки:**
- [ ] Docker-контейнеры запущены (`docker-compose up --build -d`)
- [ ] Лендинг доступен на `http://localhost:8000/`
- [ ] Кнопка «Войти» с лендинга переводит на `/login`
- [ ] Кнопка «Зарегистрироваться» переводит на `/register`
- [ ] Регистрация работодателя → попадает в `/employer`
- [ ] Регистрация студента → попадает в `/student`
- [ ] Логин работает для существующих пользователей
- [ ] Топ-студенты отображаются (если есть данные) или показывается плейсхолдер
- [ ] Навигация по всем ролевым панелям работает
- [ ] WebSocket чат открывается (если есть контакт-реквест)

## Функциональные требования

- FR-1: Маршрут `/` рендерит `LandingPage` для неавторизованных; авторизованные перенаправляются в свой кабинет
- FR-2: Маршрут `/landing` является алиасом для `/` (redirect или рендер того же компонента)
- FR-3: Хедер лендинга содержит логотип, кнопку «Войти» и «Зарегистрироваться»
- FR-4: Хедер лендинга не показывается авторизованным пользователям (у них есть свой хедер)
- FR-5: Hero-секция содержит заголовок, подзаголовок и CTA-кнопки
- FR-6: Секция преимуществ — 3-4 карточки в адаптивной сетке
- FR-7: Секция топ-студентов загружает данные из `GET /api/v1/landing/top-students`
- FR-8: Footer CTA с кнопкой регистрации
- FR-9: Все секции адаптивны (desktop + mobile)
- FR-10: Стилизация использует существующие CSS-переменные из `global.css`

## Вне скоупа

- Новые backend-эндпоинты (используем существующий `/api/v1/landing/top-students`)
- Изменения в ролевых панелях (student, employer, admin)
- Анимации при скролле (parallax, intersection observer) — только hover-эффекты
- SEO-оптимизация (meta-теги, SSR)
- Мультиязычность

## Технические соображения

- Переиспользовать CSS-переменные из `global.css` (dark theme: `--bg`, `--accent`, `--green`, etc.)
- Компонент `LandingPage` остаётся в `frontend/src/pages/landing/LandingPage.tsx`
- Хедер лендинга — новый компонент (не использует существующий `Header` из `App.tsx`, т.к. тот для залогиненных)
- Логика `HomeRedirect` в `App.tsx` упрощается: авторизованные → кабинет, неавторизованные → `LandingPage` рендерится прямо на `/`
- Frontend build output: `npm run build` → `../app/static/dist/` → FastAPI serve
- E2E тесты через Playwright MCP на `http://localhost:8000` (Docker)

## Метрики успеха

- Лендинг открывается за < 2 секунд на `/`
- Все кнопки навигации работают (Войти → /login, Зарегистрироваться → /register)
- Полный E2E-прогон проходит без ошибок (лендинг + регистрация + логин + панели)
- Нет визуальных регрессий на существующих страницах
