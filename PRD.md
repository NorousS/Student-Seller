# PRD: HH.ru Student Evaluator (Система оценки рыночной стоимости студентов)

**Версия:** 3.0
**Статус:** Final
**Область:** EdTech / HR-Tech

---

## 1. Обзор продукта

### 1.1 Описание

**HH.ru Student Evaluator** — аналитическая платформа, которая оценивает потенциальную зарплату студента на основе его учебных достижений. Система использует семантический поиск для сопоставления академических дисциплин с реальными требованиями вакансий на рынке труда (через API hh.ru).

Платформа поддерживает три роли пользователей:
- **Admin** — управление парсингом, студентами, тегами и оценками.
- **Student** — самооценка навыков, управление профилем, общение с работодателями.
- **Employer** — поиск студентов по должности, просмотр анонимизированных профилей, запрос контактов.

### 1.2 Основная проблема

Студенты часто не понимают, как их оценки и предметы в вузе конвертируются в рыночную стоимость, а вузам сложно оценить актуальность учебных программ. Работодателям трудно найти подходящих кандидатов среди студентов. Продукт решает задачу **объективизации стоимости знаний** и **сопоставления студентов с вакансиями**.

## 2. Целевая аудитория

- **Студенты:** Хотят знать свою «цену» на рынке и понимать, какие навыки приносят больше денег.
- **Методисты вузов:** Анализ соответствия курсов требованиям работодателей.
- **Работодатели (Employer):** Поиск талантливых выпускников с высоким «потенциальным чеком», анонимизированный отбор, запрос контактов.

## 3. Функциональные требования

### 3.1 Аутентификация и авторизация

- **Регистрация:** Email + пароль. При регистрации выбирается роль (admin, student, employer).
- **Авторизация:** JWT (access token 30 мин + refresh token 7 дней).
- **Роли:**
  - `admin` — полный доступ ко всем эндпоинтам.
  - `student` — профиль, самооценка, запросы на контакт, чат.
  - `employer` — поиск студентов, анонимизированные профили, запросы на контакт, чат.
- **Защита эндпоинтов:** Все API-эндпоинты (кроме auth и health) требуют JWT Bearer-токен.

### 3.2 Сбор данных (Parsing)

- **Доступ:** Только admin.

- **Интеграция с hh.ru:** Автоматический сбор вакансий по заданным ключевым словам через `api.hh.ru/vacancies`.
- **Фильтрация по опыту:** Поддержка фильтра по уровню опыта (`noExperience`, `between1And3`, `between3And6`, `moreThan6`).
- **Обработка навыков:** Извлечение Key Skills из каждой вакансии, сохранение в PostgreSQL с дедупликацией, и параллельная индексация в Qdrant через эмбеддинги.
- **Зарплатная статистика:** Для расчётов используются только вакансии с зарплатой в рублях (`salary_currency = "RUR"`).

### 3.3 Математическая модель расчёта (Valuation Engine)

Оценка строится на четырёх этапах:

**Этап 1: Фильтрация по специальности**

Перед оценкой система семантически сопоставляет указанную специальность студента с поисковыми запросами вакансий (поле `search_query`). Используются только вакансии, чей запрос прошёл порог:

$$\mathrm{similarity}(\mathrm{specialty}, \mathrm{search\_query}) \geq 0.7$$

Это гарантирует, что оценка основана на профильных вакансиях.

**Этап 2: Семантический поиск навыков**

Для каждой дисциплины студента:
1. Название дисциплины преобразуется в вектор через `nomic-embed-text` (768 измерений).
2. �� Qdrant ищутся `top_k` (1–20, по умолчанию 5) ближайших навыков по косинусному расстоянию.
3. Отсеиваются навыки с `similarity < 0.6` (порог настраивается).
4. Отсеиваются навыки, встречающиеся менее чем в 3 вакансиях (`MIN_TAG_COUNT = 3`).
5. Удаляются вручную исключённые пользователем навыки.

**Этап 3: Вес навыка ($W_i$)**

Вес каждого навыка определяется тремя факторами:

$$W_i = \mathrm{similarity}(d, s_i) \cdot \ln(1 + \mathrm{count}_i) \cdot k_g$$

| Компонент | Описание |
|-----------|----------|
| $\mathrm{similarity}(d, s_i)$ | Косинусное сходство между вектором дисциплины $d$ и навыка $s_i$ (0–1) |
| $\ln(1 + \mathrm{count}_i)$ | Натуральный логарифм от числа вакансий с этим навыком. Сглаживает выбросы — навык из 1000 вакансий не в 1000 раз важнее навыка из 10 |
| $k_g$ | Коэффициент оценки: «5» = 1.0, «4» = 0.85, «3» = 0.75 |

**Этап 4: Итоговая оценка**

Единая взвешенная средняя по **всем навыкам всех дисциплин** студента:

$$\mathrm{estimated\_salary} = \frac{\sum_{i} (\mathrm{avg\_salary}_i \cdot W_i)}{\sum_{i} W_i}$$

где $\mathrm{avg\_salary}_i$ — средняя зарплата по вакансиям, содержащим навык $i$.

**Уровень уверенности (Confidence):**

$$\mathrm{confidence} = \frac{\mathrm{matched\_disciplines}}{\mathrm{total\_disciplines}}$$

- `matched_disciplines` — число дисциплин, для которых нашёлся хотя бы один навык выше всех порогов.
- Значение от 0.0 до 1.0.

### 3.4 Профиль студента

- **Доступ:** Только student (владелец).
- **«О себе»:** Текстовое поле, видимое работодателю только после принятия запроса на контакт.
- **Фото профиля:** Загрузка JPEG/PNG (до 5 МБ). Если фото нет — заглушка.
- **Самооценка навыков:** Студент может оценить свои дисциплины (аналог админского функционала), но **не может** парсить вакансии.

### 3.5 Панель работодателя

- **Доступ:** Только employer.
- **Поиск студентов по должности:** Ввод названия должности → система подбирает студентов с сортировкой по confidence и estimated_salary.
- **Анонимизированные профили:** Работодатель видит дисциплины, оценки, фото, estimated_salary — **без ФИО, группы и раздела «о себе»**.
- **Запрос на контакт:** Работодатель отправляет запрос → студент принимает/отклоняет → при принятии открываются «о себе» и чат.

### 3.6 Чат

- **Транспорт:** WebSocket (`/ws/chat/{contact_request_id}?token=JWT`) с REST fallback.
- **Условие:** Чат доступен только после принятия запроса на контакт студентом.
- **Персистентность:** Все сообщения сохраняются в PostgreSQL (таблица `messages`).
- **История:** REST-эндпоинт для получения истории с пагинацией.

### 3.7 Интерфейс (React SPA)

#### Страницы авторизации
- Логин (email + пароль).
- Регистрация (email, пароль, роль, доп. поля в зависимости от роли).

#### Админ-панель (role: admin)
- Выбор студента из списка, ввод специальности, фильтры, графики, таблица навыков.

#### CRUD студентов (admin)
- Создание студента (ФИО, группа), добавление дисциплин с оценками.

#### Парсинг вакансий (admin)
- Поисковый запрос, фильтр по опыту, статистика, лог.

#### Навыки и теги (admin)
- Поиск, pie-chart, таблица тегов.

#### Панель студента (role: student)
- **Профиль:** ФИО, группа, «о себе», фото.
- **Навыки:** Самооценка дисциплин (оценки 3/4/5).
- **Запросы на контакт:** Входящие запросы от работодателей, принять/отклонить.
- **Чат:** Переписка с работодателями (после принятия запроса).

#### Панель работодателя (role: employer)
- **Поиск:** Ввод должности → список анонимизированных студентов.
- **Профиль студента:** Анонимизированные данные, кнопка «Запросить контакт».
- **Запросы:** Список своих запросов на контакт.
- **Чат:** Переписка со студентами.

#### Общие особенности UI
- Тёмная тема (GitHub-стиль).
- Адаптивная сетка.
- Все графики — Chart.js (Bar, Radar, Pie).
- Стек: Vite + React + TypeScript.

## 9. Риски и ограничения

### Риски

| Риск | Митигация |
|------|-----------|
| Низкое качество данных (отсутствие зарплат в вакансиях) | Учитываются только вакансии с указанной зарплатой в рублях; навыки с < 3 вакансий отсеиваются |
| Семантический шум (неправильное сопоставление предметов) | Ручная фильтрация навыков пользователем + пороги сходства (0.6 для навыков, 0.7 для специальности) |
| Таймаут Ollama при первой загрузке модели | Ollama автоматически скачивает модель при первом запросе (таймаут 300 сек) |

| Утечка JWT-секрета | Использовать переменные окружения, не коммитить в код |

### Текущие ограничения

- Нет пагинации для списка студентов.
- Нет эндпоинта удаления студента.
- Нет кэширования эмбеддингов (пересчёт при каждом запросе оценки).
- Нет rate-limiting на эндпоинте парсинга.
